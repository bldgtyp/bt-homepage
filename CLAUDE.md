# BLDGTYP Homepage - Agent Instructions

This repo is the source for `bldgtyp.com`, the public homepage for BLDGTYP,
LLC. BLDGTYP is a Brooklyn-based Passive House consulting and design practice
serving mostly NYC, NY State, northern NJ, Long Island, the Hudson Valley, and
the Berkshires/MA.

Work in this repo should be direct, careful, and narrow. The deployed site is
small, static, and public-facing; accidental changes to generated output,
deploy behavior, or existing Dreamhost content can have outsized effects.

## Owner Context

- Primary user: Ed May, licensed architect, CPHC (PHI and Phius), former
  carpenter/builder, BLDGTYP co-founder.
- Partner: John Mitchell.
- Domain expertise: envelope design, thermal bridging, assemblies,
  hygrothermal/moisture risk, PHPP/WUFI-Passive, PHI/Phius certification.
- Preferred tone: terse, technical, evidence-based. Use Passive House terms
  naturally: U-value, psi-value, SHGC, ACH50, iCFA, WUFI, PHPP, Flixo, Dartwin.
- If discussing commercial/multifamily mechanical systems, explain more and
  bridge from residential mechanical intuition.

## Repo Shape

Only `site/` is deployed to Dreamhost.

```text
bt-homepage/
|-- site/                     # deployable static site
|   |-- index.html            # generated by Hugo
|   |-- about/                # generated by Hugo
|   |-- learn/                # generated by Hugo
|   |-- ma/                   # generated by Hugo
|   |-- nyc-thermal-bridging/ # generated by Hugo
|   `-- assets/
|-- hugo/                     # Hugo source for all pages
|   |-- hugo.toml
|   |-- content/
|   |-- data/videos.yaml
|   `-- layouts/
|-- research/                 # source research, transcripts, synthesis
`-- .github/workflows/deploy.yml
```

Important boundaries:

- `hugo/` is the source of truth for all page markup, shared styles, shared
  scripts, nav, footer, and video-library data.
- `site/` is generated deploy output plus static assets. Do not hand-edit
  generated HTML under `site/`; edit `hugo/` sources and rebuild.
- `hugo/hugo.toml` publishes into `../site`, including the homepage.
- `research/` is supporting material, not deploy output. It holds the
  marketing research and copy decisions that govern site voice and structure —
  see "Marketing Research & Copy Guidelines" below.
- `.DS_Store`, `hugo/.hugo_build.lock`, and `hugo/resources/` are ignored.

## Commands

Run the Hugo hot-reload dev server:

```bash
cd hugo
hugo server --bind 127.0.0.1 --baseURL http://127.0.0.1:1313/
```

Open:

```text
http://127.0.0.1:1313/
```

Build generated deploy output:

```bash
cd hugo
hugo
```

Serve generated deploy output for exact static-output review:

```bash
cd site
python3 -m http.server 8000 --bind 127.0.0.1
```

Open:

```text
http://127.0.0.1:8000/
```

Expected Hugo version in current local environment:

```text
hugo v0.124.1+extended darwin/arm64
```

There is no package manager or JS build step for the current site. Do not add
Node/Vite/React tooling unless the user explicitly asks for a larger rebuild.

## Local Preview Notes

In the current Codex desktop environment, `hugo server` on
`127.0.0.1:1313` is the normal editing preview path. Serving generated `site/`
on `127.0.0.1:8000` is useful when verifying exact deploy output.

Other sandboxed Claude/Cowork environments may not expose VM localhost to the
user's Chrome browser. If a browser cannot reach the local server, fall back to
opening `site/index.html` directly or ask the user to open/refresh the page and
share a screenshot. `curl` is still useful for checking response codes and
asset paths.

## Editing Rules

- Keep edits surgical and scoped to the requested change.
- Before processing a directory, check for `.index.md`, `_index.md`, or
  `.instructions.md`.
- For large files, run `wc -l` first. If over 2000 lines, read in chunks.
- Preserve user changes in the working tree. Do not revert or rewrite unrelated
  local diffs.
- Use `rg` or `rg --files` for search.
- Prefer `pnpm` for TypeScript repos generally, but this repo currently has no
  JS package setup.
- Do not install packages into system Python. This repo should not need Python
  packages for normal static serving.
- Use ASCII in code/docs unless the touched file already clearly uses Unicode
  and the character is intentional for public copy.

The `.graph-paper` base class is required for positioning context; the
modifier classes alone are not enough.

When changing pages:

- Edit Hugo source files, not generated HTML in `site/`.
- Use `hugo server` for hot-reload development.
- Rebuild with `cd hugo && hugo` before final verification.
- Preserve the external branding token contract where possible.
- Keep typography, colors, radii, and component behavior aligned with the
  design-system tokens when possible.
- `data-theme="light"` / `data-theme="dark"` on `<html>` controls theming.
- The Google Fonts import for Outfit and JetBrains Mono belongs in each
  consuming page's `<head>`.
- Verify key viewports after visual changes: mobile, tablet, desktop.
- Check that nav, CTAs, theme behavior, and asset paths work from the static
  server root after rebuilding.
- Use real project/service language. Avoid generic agency marketing copy.

The `/learn/` page is generated from `hugo/data/videos.yaml` plus Hugo
templates. Preserve the source data structure unless the user asks for a schema
change.

## Marketing Research & Copy Guidelines

The site's voice, structure, and positioning are grounded in a body of research
under `research/` — do not treat homepage copy or page design as a blank slate.
When editing any page content, writing copy, choosing headings/labels, or
designing page elements (hero, cards, CTAs, section order), consult these first
and align with them. The `Content Voice` section below is the short summary;
these files are the reasoning behind it.

Source of truth, in dependency order (see `research/.index.md` and
`research/synthesis/.index.md` for full maps):

- `research/synthesis/marketing-rules.md` — master rules distilled from 38
  "Marketing Passivhaus" episodes, each cited to the guest who voiced it. The
  authoritative field guide for language to use/avoid, audience framing, proof,
  and pitfalls. `marketing-rules.html` is a rendered copy; edit the `.md`.
- `research/synthesis/homepage-alignment-and-business-model.md` — audits the
  current `site/index.html` against those rules; explains what to keep, fix, or
  cut and why.
- `research/synthesis/homepage-copy-redline.md` — the concrete section-by-section
  copy edits keyed to `site/index.html`. Check its STATUS block before writing
  homepage copy: it records what is already implemented vs. staged/blocked.
- `research/synthesis/positioning-decisions.md` — the decisions log. Treat
  DECIDED items (e.g. D1: B2B / design-professional audience, not
  direct-to-homeowner) as binding constraints; do not reopen them casually.
  Note which decisions are still open before making a call that depends on them.
- `research/transcripts/` and `research/episode-notes/` — raw source (38 guests)
  behind the rules; consult only when you need the original context for a rule.

Practical rules:

- Voice the site for design professionals (architects, developers, builders,
  certification teams); homeowners "ride along" but are not the primary buyer
  (D1).
- Lead with concrete outcomes and services, not jargon or generic
  sustainability language — per the marketing rules.
- Do not contradict a DECIDED positioning call or an implemented redline item
  without flagging it to Ed first.
- When these guidelines and older copy in `site/index.html` disagree, surface
  the conflict rather than silently picking one.

## Content Voice

BLDGTYP positioning:

- Passive House consulting and design.
- Strong technical delivery for architects, developers, owners, builders, and
  certification teams.
- Core work: design analysis, certification, thermal bridge simulation, energy
  modeling, assemblies, airtightness, and documentation.
- Typical projects: single-family/townhouse retrofits, second homes,
  affordable multifamily, high-performance building consulting, and thermal
  bridge simulations.
- Geography: NYC metro, NY State, northern NJ, Long Island, Hudson Valley, and
  Massachusetts/Berkshires when relevant.

Style:

- Direct, specific, and technically credible.
- Prefer concrete service language over vague sustainability language.
- Avoid overclaiming. Do not imply BLDGTYP is the architect of record unless
  the copy explicitly supports that.
- Use "Passive House" for the general standard, and PHI/Phius only when the
  distinction matters.
- Do not invent projects, metrics, certifications, client names, or awards.

## Verification Expectations

For content-only or template edits:

```bash
cd hugo && hugo
```

Then check the affected route with `hugo server`, browser inspection, or
`curl -I` against generated `site/` output.

For visual/static asset edits:

- Use `hugo server` while editing.
- Rebuild generated `site/`.
- Check `/`, changed anchors, and changed assets.
- Use browser inspection or screenshots for layout-sensitive work.

For deploy workflow edits:

- Inspect `.github/workflows/deploy.yml`.
- Keep server-only content protection explicit.
- Do not add delete/removal flags.
